# 通知メッセージのフォーマット

SKILL.md のステップ4「Slack投稿」で本文を組み立てる際の**書式の正本**。投稿前に必ずこのファイルに従うこと。

**2段構成**で送る。**親投稿**（チャンネルに投稿し通知を発火・催促）＋ その **子スレッド返信**（トラッカー別→担当者別の明細）。

## ⚠️ 最重要の落とし穴: メンションは見出し行に置かない

Slack は **`#` / `##` / `###` の見出し行ではメンション記法 `<@U…>` を展開しない**（生の `<@U03N96QEZ>` がそのまま表示される。実機で確認済み）。
→ **担当者のメンションは必ず通常のテキスト行に置く**こと。トラッカー見出しだけ `##` を使う（メンションを含まないので問題ない）。

## 親投稿（チャンネル投稿。通知を発火・催促）

未対応が1件以上あるとき:

```markdown
:rotating_light: *未対応チケットが溜まっています。対応 / ステータス更新をお願いします* 🙏

合計 {総件数}件（障害 {X} / ご質問・調査依頼 {Y}）

担当者別の未対応数:
<@{User ID}> {n}件
<@{別の User ID}> {m}件
```

- **催促トーン**: 先頭に 🚨（`:rotating_light:`）＋「溜まっています／対応をお願いします」。"被催されている" 感覚を出すのが狙い。
- **担当者別の未対応数**: 通常テキスト行で `<@{User ID}> {件数}`（通常行なのでメンションは展開され `@西尾 誠 10件` と表示される）。**未対応が1件以上ある担当者だけ**を載せる（0件の人は載せない・メンションしない）。並びは `assignees` の config 順。
- 件数 = その担当者の全トラッカー合計（障害＋ご質問…）。
- 明細はここには載せない（子スレッドに載せる）。

**全トラッカー0件の場合**（子スレッドは送らない。催促もしない）:

```markdown
本日の未対応チケット: 0件 🎉
```

## 子スレッド（親の thread_ts に投稿。トラッカーごとに1通）

```markdown
## 障害 (X件)

<@{担当者の User ID}> (N件)
- <{KINTONE_BASE_URL}/k/{app_id}/show#record=12345|[#12345] 題名1>
- <{KINTONE_BASE_URL}/k/{app_id}/show#record=12346|[#12346] 題名2>

<@{別の担当者の User ID}> (M件)
- ...
```

### 書式ルール

- **トラッカー見出し**: `## {トラッカー名} ({件数})`（`##` 見出し。メンションを含まないのでOK）。**0件のトラッカーは出さない**。
- **担当者見出し**: **通常テキスト行**の `<@{担当者の User ID}> ({件数})`。⚠️ **`###` などの見出し行にしないこと**（見出し行ではメンションが展開されず生表示になる）。User ID は `assignees` の email を `slack_search_users` で解決して得る。**解決できない email は `{名字}さん ({件数})` の名字テキストにフォールバック**（メンションなし）。
- **「その他」グループ**: `assignees` に含まれない担当者だけのチケットは末尾に `その他 ({件数})`（通常行・メンションなし）。※通常は Person フィルタにより発生しない（後述）。
- 各チケット: **1行**で Slack のリンク記法 `- <{チケットURL}|[#{レコード番号}] {題名}>`。タイトル自体をリンクにし、**生 URL は出さない**。
  - チケット URL は `{KINTONE_BASE_URL}/k/{app_id}/show#record={レコード番号}`（base は環境変数から）。
  - 題名内の `@xxx`（`<@U…>` / `@表示名` 等のメンション記法）は**削除してから**リンク化（誤通知・表示崩れ防止）。題名内の `|` は最初の `|` が区切りになるだけで表示は崩れない。
- 件数: トラッカー見出し=そのトラッカーの全件、担当者=そのグループ内の件数。
- **件数の自己検査（送信前必須）**: トラッカー件数 = 全担当者グループ件数の合計 = `get-records` の `totalCount`。3者が一致しないときは送信せず失敗扱い（数え間違いのまま送らない）。
- 並びは `assignees` の config 順（項目があるグループのみ）、最後に「その他」。グループ内のチケットは Kintone 取得順（更新日時 desc）のまま。

### 「その他」が通常発生しない理由
`get-records` を `Person inValues = assignees(email)` で絞っているため、取得レコードは必ず assignee を1人以上含む。したがって「命中者なし＝その他」は通常起こらない（防御的に規則だけ残している）。

## 分割ルール（メッセージ長対策・確定的）

Slack の1メッセージは約 5000 文字まで。**文字数を推定して判断しない**（推定は実行ごとにブレるため）。**件数（配列長）で機械的に判定**する。

- 子スレッドは**トラッカーごとに1通**が基本。
- **1通あたり 25 件**を上限とする。担当者グループを `assignees` 順に詰めていき、次のグループを加えると 25 件を超える場合は、そこで区切って**続きの通**を送る。続きの通の見出しは `## {トラッカー名} (続き)`。
- **グループの途中では絶対に切らない**（担当者グループは分割の最小単位）。
- ただし**単一グループが 25 件を超える**場合のみ、そのグループを 25 件ごとに分割し、続きの担当者行は `<@{User ID}> (続き)`（これも通常テキスト行）とする。
- 分割しても、親投稿のトラッカー件数は**全通の合計**（＝ `totalCount`）であること。

例: あるトラッカーが 西尾9 / 張5 / 川下3 / 澤松23 のとき → 1通目「西尾9+張5+川下3=17件」、2通目「澤松23件（続き）」。
